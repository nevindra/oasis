# Migration: 0.18 → 0.19 (Plan C — Workflow as Agent)

This release finishes making `Workflow` a first-class `core.Agent` peer of `LLMAgent` and `Network`. The bulk of the work landed in Plan A (`Workflow.Execute(ctx, task, ...RunOption)` — a `core.Agent` already at the end of 0.17). Plan C does the cleanup: drop the duplicate `ToolStep` API, trim `WorkflowContext`'s public surface, and ship composability tests proving Workflow⇄Network compose freely.

## Quick map

| If you wrote… | Change to… |
|---|---|
| `workflow.NewWorkflow(name, desc, opts...)` | `workflow.New(name, desc, opts...)` (already wired in 0.18 — umbrella `oasis.NewWorkflow` keeps its name) |
| `workflow.ToolStep("name", tool, "tool_name", opts...)` | `workflow.AgentStep("name", agent.New("inner", "...", p, agent.WithTools(tool)))` |
| `oasis.ToolStep(...)` | Same: wrap the tool in a one-tool `agent.New(...)` and use `workflow.AgentStep`. The `oasis.ToolStep` re-export is gone. |
| `workflow.NewWorkflowContext(task)` | *(no replacement — see "Why" below)* |
| `result.Context.Get("key")` on a `WorkflowResult` | `result.Steps["stepname"].Output` — but only if the step wrote to its Output. If you used `WorkflowContext` as a side-channel KV across step boundaries, refactor your steps to publish to their own outputs. |

## Why

The 0.18 surface had three things that worked against "Workflow is just an Agent":

1. **`ToolStep`** was a second way to do what `AgentStep` already did with a one-tool LLMAgent. The framework's "one shape per task" principle says delete the duplicate. The internal YAML/JSON definition path (`workflow.FromDefinition`) keeps an unexported helper to preserve its direct-tool-call semantics — user-facing code uses `AgentStep`.
2. **Public `NewWorkflowContext`** suggested users should construct a context. They shouldn't — the framework owns its lifecycle. The constructor was vestigial from an earlier API shape where tests outside the workflow package needed to drive `ResumeData`.
3. **Public `WorkflowResult.Context`** let callers peek at the workflow's shared state after a run, which made it hard to evolve workflow internals without breaking callers. Step outputs (`result.Steps[name].Output`) are the documented contract.

`WorkflowContext` itself stays public because `StepFunc` still takes it as a parameter — hiding it would require a `StepFunc` redesign, which is deferred to a separate, larger workflow revamp. Treat it as **step-local**: read/write it from inside the step you're currently in; don't reach for it across step boundaries.

## What's NOT changed in this release

- `Step`, `AgentStep`, `ForEach`, `DoUntil`, `DoWhile` keep their shapes.
- `WorkflowContext`'s public methods (`Get`, `Set`, `Input`, `Resolve`, `ResolveJSON`) are unchanged.
- `WorkflowResult.Status`, `Steps`, `Usage` are unchanged.
- `Suspend` / `ErrSuspended` in workflow are unchanged (covered by Plan D).
- The `oasis.NewWorkflow` umbrella name is unchanged (now delegates to `workflow.New`).

## What's still ahead

- **Plan D** — middleware unification, skills trim, stream consolidation, suspend cleanup, `oasis.go` full trim, `workflow.go` aliases cleanup.
- **Future workflow redesign** (its own spec) — typed `AgentTask[In, Out]`, step-local typed inputs/outputs, possible removal of `WorkflowContext` entirely.

## Examples

### Three-step workflow with a tool step — before and after

**Before:**
```go
wf, err := workflow.NewWorkflow("research", "Research workflow",
    workflow.AgentStep("plan", planner),
    workflow.ToolStep("fetch", httpTool, "fetch_url"),
    workflow.AgentStep("write", writer),
)
```

**After:**
```go
fetcher := agent.New("fetcher", "Fetches URLs", p, agent.WithTools(httpTool))
wf, err := workflow.New("research", "Research workflow",
    workflow.AgentStep("plan", planner),
    workflow.AgentStep("fetch", fetcher),
    workflow.AgentStep("write", writer),
)
```

### Workflow as a Network child — same shape as any agent

```go
research := workflow.New("research", "Research workflow", /* steps */)
team := network.New("team", "Research team", routerP, research, fastResponder)
```

### Network as a Workflow step's agent — same shape as any agent

```go
subnet := network.New("subnet", "Subnet team", subRouterP, a, b)
wf := workflow.New("paper", "Paper workflow",
    workflow.AgentStep("research", subnet),
    workflow.AgentStep("writeup", writer, workflow.After("research")),
)
```

Both compose because `*Workflow` and `*Network` satisfy `core.Agent`. See `workflow/composition_test.go` (`TestWorkflowInNetwork`, `TestNetworkInWorkflow`) for the verified scenarios.

### Reading step results — before and after

**Before:**
```go
result, _ := wf.Execute(ctx, task)
draft, _ := result.Context.Get("write.output")
```

**After:**
```go
result, _ := wf.Execute(ctx, task)
draft := result.Steps["write"].Output
```

## Divergences from the design spec

- `WorkflowContext` is kept exported because `StepFunc` takes it as a parameter; tightening to a step-local scope requires a `StepFunc` redesign that lives in a separate spec.
- `workflow/workflow.go` local type aliases (`type AgentTask = core.AgentTask` etc.) are not touched — they're an internal convenience analog to the deleted `agent/aliases.go` and don't block the design goals. Plan D will revisit.
- `workflow.ArgsFrom` is kept (still consumed by the internal YAML/JSON tool-call path). Doc comments were updated to reflect that it's a tool-call-step option, not a `ToolStep`-specific one.
