## Theme

Tool authors stop writing JSON Schema. The schema is derived from the input struct via reflection at registration time. One source of truth (the Go struct); one method to write (`Definition() ToolMeta`); one method to override when reflection isn't enough (`JSONSchema() json.RawMessage` on the input type).

## Goals

1. Replace the hand-written `Parameters: json.RawMessage(...)` block in every tool's `Definition()` with reflection over the input type `In`.
2. Shrink the `Tool[In, Out]` interface from 3 methods (`Name`, `Definition`, `Execute`) to 2 (`Definition`, `Execute`).
3. Push schema-shape errors to *registration time* (server startup) instead of *LLM-call time* (silent JSON parse failures or LLM-confusion).
4. Lock in a minimal supported-types table and tag vocabulary as the permanent contract.

## Non-goals (deferred to later phases)

- **`min`/`max`/`pattern`/`format` tags.** Provider support is uneven; LLMs often emit out-of-range values anyway. Add as additive future tags only when real demand surfaces.
- **`Enumer` interface as alternative to enum tags.** Cleaner Go for typed-string enums; defer until the tag-based form's limitations bite real users.
- **Examples / few-shot prompts in `ToolMeta`.** Out of scope â€” different concern.
- **Description from struct doc comments.** Would require `go:generate` or AST parsing; out of scope for a runtime-reflection release.
- **Type-name â†’ tool-name inference.** Magical, brittle â€” type renames silently change the LLM-facing name.
- **Output schema derivation.** Only inputs in 1.5. Outputs are framework-internal (marshaled by `Erase`'s `erasedTool`).

## Out of scope, already shipped in Phase 1

- `StreamingTool[In, Out]` (1.2.i) shipped in commit `5f556fb`. Phase 1.5 updates its embedded `Tool[In, Out]` shape automatically.

## Design decisions

Each decision below documents the chosen approach, the rationale, and the alternatives considered.

### 1. Reflection is required; `Definition()` is removed from the user interface

**Today:**

```go
type Tool[In, Out any] interface {
    Name() string
    Definition() ToolDefinition  // Parameters is hand-written JSON Schema
    Execute(ctx context.Context, in In) (Out, error)
}
```

**Designed:**

```go
type Tool[In, Out any] interface {
    Definition() ToolMeta
    Execute(ctx context.Context, in In) (Out, error)
}

// ToolMeta is the metadata an author writes by hand. The schema is derived
// from In by reflection inside Erase().
type ToolMeta struct {
    Name        string
    Description string
}
```

**Why required (not additive):** an additive design (keep `Definition() ToolDefinition` as an escape hatch alongside reflection) means two paths to teach, two paths to maintain, and ongoing pressure to add features to the inferior path. Required reflection is one way to write a tool; the escape hatch is a *different* mechanism (a method on the input type â€” see Decision 3) with a different name.

**Why one method instead of three (`Name`/`Description`/`Execute`):** writing a single object expresses metadata as a cohesive thing rather than three unrelated calls. Matches `ToolDefinition` ergonomics today, just with the `Parameters` field removed.

**Why a new `ToolMeta` type rather than reusing `ToolDefinition`:** `ToolDefinition` has a `Parameters` field. If we reused it, an author could set `Parameters` directly, opening the "what wins, reflection or explicit?" question. A separate type makes the no-schema rule structural â€” no field exists for the author to fill.

`core.ToolDefinition` stays exactly as it is today; it is now produced *by* `Erase()` rather than written by the user.

**Alternatives rejected:**
- *Keep `Definition()` as opt-out, reflection as opt-in.* Means two paths forever; defeats the design intent.
- *Three separate methods (`Name`/`Description`/`Execute`).* Author preference confirmed for the single-object shape.
- *Comment-derived description (`//tool:"data_parse" describe:"..."`).* Go has no runtime access to comments; requires `go:generate`. Out of scope for this release.

### 2. Supported-types table

The table below is the **permanent contract** post-v1.0. Every row is a forever promise.

| Go type | JSON Schema |
|---|---|
| `bool` | `{"type":"boolean"}` |
| `int`, `int8`, `int16`, `int32`, `int64` | `{"type":"integer"}` |
| `uint`, `uint8`, `uint16`, `uint32`, `uint64` | `{"type":"integer"}` |
| `float32`, `float64` | `{"type":"number"}` |
| `string` | `{"type":"string"}` |
| `[]T` | `{"type":"array","items":<schema of T>}` |
| `[]byte` | `{"type":"string"}` (encoding/json marshals as base64 already) |
| `map[string]T` | `{"type":"object","additionalProperties":<schema of T>}` |
| `struct` | `{"type":"object","properties":{...},"required":[...]}` (recursive) |
| `*T` | schema of T; field NOT in `required` |
| `any` / `interface{}` | `{}` |
| `json.RawMessage` | `{}` (falls out of `[]byte` rule; documented separately for clarity) |
| `time.Time` | `{"type":"string","format":"date-time"}` |

**Rejected (panic at `Erase()` with a clear error):**
- `complex64`, `complex128` â€” not JSON
- `chan`, `func`, `unsafe.Pointer` â€” not JSON
- `map[K]V` where K is not `string` â€” JSON object keys must be strings
- Recursive types (struct that contains itself directly or via pointer) â€” push to `JSONSchema()` escape hatch
- `byte` as a single struct field (it's `uint8`, ambiguous against the special `[]byte` rule) â€” use `uint8` explicitly if that's intended; `[]byte` works as expected

**Optionality rules**: a field is NOT in `required` if **either** signal is present (signals are additive, not exclusive):
- Field type is a pointer (`*T`)
- Field's `json` tag contains `,omitempty`

**Embedded struct rules**: follow `encoding/json`'s embedding rules. Anonymous fields flatten into the outer schema unless renamed via the `json` tag. Conflicts (same JSON name from two embedded structs) panic at `Erase()` â€” same Go rule, just enforced at schema-derivation time.

**Why this table is small:** every type accepted is a contract we keep forever. Numeric widths all collapse to `integer`/`number` because LLMs don't honor numeric range constraints reliably â€” exposing `int8`'s implicit `-128..127` range to the schema would be a lie. Same reasoning for `[]byte` â†’ bare `"type":"string"` instead of `{"type":"string","contentEncoding":"base64"}` (inconsistent provider support).

**Alternatives rejected:**
- *Accept only `int`/`int64`/`float64`.* Forces idiomatic Go code that uses `int32`/`uint8`/etc. to convert at the boundary. Annoying for no real benefit.
- *Emit numeric ranges (`{"type":"integer","minimum":-128,"maximum":127}` for `int8`).* Schema is a lie when providers ignore the range; LLMs emit out-of-range values regardless.
- *Reject `time.Time`.* Common pattern in tools; stdlib type with well-known wire format; convenience is worth the special case.

### 3. Escape hatch: `JSONSchema()` method on the input type

When reflection cannot express what a tool needs (`oneOf`, conditional `required`, recursive shapes, provider-specific features), the *input type* opts itself out:

```go
type ExoticIn struct {
    Mode string          `json:"mode"`
    Args json.RawMessage `json:"args"`
}

// JSONSchema overrides reflection. The struct's Go shape still has to be
// json.Unmarshal-able into something Execute can use; this method only
// affects the schema shown to the LLM.
func (ExoticIn) JSONSchema() json.RawMessage {
    return json.RawMessage(`{"oneOf":[...]}`)
}
```

The reflector detects this via a type assertion at `Erase()` time. If `In` implements `SchemaProvider`, that override is used; otherwise reflection proceeds.

**The interface lives in `core/`:**

```go
type SchemaProvider interface {
    JSONSchema() json.RawMessage
}
```

**Why a method on the input type (not on the Tool):**
- Separates "what the LLM sees" from "what the tool author writes day-to-day". The `Definition() ToolMeta` method stays minimal even for tools using the escape hatch.
- Mirrors `encoding/json.Marshaler` and `database/sql.Scanner` â€” standard Go pattern for "implement a method to override default behavior."
- Lets the input type be unit-tested independently: `ExoticIn{}.JSONSchema()` returns bytes a JSON Schema validator can check.

**Why not `json.RawMessage` as the In type (a "use the existing escape valve" alternative):** that produces a fully empty `{}` schema, which is a *surrender* â€” the LLM has no guidance about what to send. That's right for "I want truly arbitrary input" but wrong for "I need a precise schema that reflection can't generate." Note: `json.RawMessage` as the In type still works (the supported-types table emits `{}`), but it's a degenerate case, not the canonical escape hatch.

### 4. Tag vocabulary (minimal)

| Tag | Example | Notes |
|---|---|---|
| `json:"name,omitempty"` | `json:"limit,omitempty"` | stdlib; honored verbatim |
| `describe:"..."` | `describe:"max records to return"` | free-text description for the LLM |
| `enum:"a,b,c"` | `enum:"csv,json,jsonl"` | comma-separated; **string fields only** |

**That's the whole vocabulary.** Three tags; two of them stdlib.

**Enum limitations:**
- String fields only. Integer enums (`enum:"1,2,3"`) require the `JSONSchema()` escape hatch.
- Comma is the delimiter. Enum values containing commas require the escape hatch. (Real-world enum values almost never contain commas.)

**Explicitly NOT in 1.5** (all available via `JSONSchema()` escape hatch):
- `min`/`max` (numeric ranges)
- `minLength`/`maxLength` (string lengths)
- `pattern` (regex)
- `format` (uuid, email, uri â€” except for `time.Time` â†’ `date-time` which is automatic)
- `default`
- `minItems`/`maxItems` (array bounds)

**Future-compatibility:** new tags can be added in non-breaking releases. Old code is unaffected by tag additions because old code doesn't use the new tags.

**Alternatives rejected:**
- *Ship `min`/`max`/`pattern`/`format` in 1.5.* Permanent contract; provider enforcement is inconsistent so the schema's promise is weaker than it looks; add later if real demand surfaces.
- *`Enumer` interface (`func (T) Enum() []string`) instead of `enum:"..."` tag.* Cleaner for typed-string enums (`type DataFormat string`) but requires defining a typed-string per enum field. The tag form covers ad-hoc enums for less ceremony. Both can coexist in a future release if `Enumer` becomes desired.

### 5. Error model: panic at `Erase()`

`Erase[In, Out](t)` signature is unchanged. On unsupported type, embedded-field conflict, or recursive type, **panic** with a clear, actionable message:

```
panic: oasis.Erase[ParseInput, ParseOutput]: field 'Limit' has unsupported type complex64
(supported integer types: int, int8, int16, int32, int64, uint, uint8, uint16, uint32, uint64)
```

**Why panic, not error return:**
- Reflection failures are *registration-time* problems. They surface at server startup, where panic is the right escalation â€” fail loudly, fail before serving requests.
- Matches Go convention for type-related setup failures: `regexp.MustCompile`, `template.Must`, `json.Marshal` on unsupported types.
- Returning `(AnyTool, error)` would force every call site (currently `oasis.Erase(&MyTool{})`) to handle the error variable explicitly â€” verbose at every registration without buying real safety, since a tool whose schema can't be derived isn't usable anyway.

**Panic vs. process-startup behavior:** Go's `panic` during package init propagates and aborts the program. For lazily-registered tools (created during request handling â€” uncommon but possible), the panic surfaces in that request's handler. This matches today's behavior for any panic during tool dispatch.

**Error message contract:** every panic message includes:
- The full generic signature (`oasis.Erase[X, Y]`)
- The field path (for nested cases, e.g., `field 'Where[].Op' has...`)
- The offending Go type
- The supported alternatives for that schema kind

### 6. Migration: every existing tool migrates atomically

This is a breaking minor bump in the v0.x line, like Phase 1. The `Tool[In, Out]` interface change is a source break â€” `Definition() ToolDefinition` is no longer a valid implementation.

**In-repo tools that migrate** (enumerated from the codebase):
- `tools/http/http.go` â€” 1 tool (`http_fetch`)
- `tools/data/data.go` â€” 4 tools (`data_parse`, `data_filter`, `data_aggregate`, `data_transform`)
- `sandbox/tools.go` â€” 2+ tools
- `skills/tool.go` â€” 4 tools
- Built-ins in `agent/`: `askUserToolDef`, `executePlanToolDef`, `spawnAgentToolDef`. These build `ToolDefinition` values directly (not through `Tool[In, Out]`), so they migrate to the new code path slightly differently â€” likely by computing schemas inline or via a small helper that uses the same reflection internals.

**External (user) tools:** users of the `oasis` umbrella with a custom `Tool[In, Out]` implementation get a compile error pointing at `Definition() ToolDefinition` no longer satisfying the new interface. They rename `Name()`+`Definition()` into a single `Definition() ToolMeta`, add `describe`/`enum` tags to their input struct fields, and delete the `Parameters: json.RawMessage(...)` block. Mechanical.

**The release notes / migration guide** will include the side-by-side examples already worked through in brainstorming.

## Implementation sketch

(Final shape gets settled in the writing-plans phase; this is just enough to confirm the design is implementable.)

**New file `core/schema.go`:**

```go
// SchemaProvider is the optional opt-out for reflection. Input types that
// implement this method provide their own JSON Schema.
type SchemaProvider interface {
    JSONSchema() json.RawMessage
}

// deriveSchema computes the JSON Schema for In. Called once per Erase[In, Out].
// Panics with a descriptive message on unsupported types or recursion.
func deriveSchema[In any]() json.RawMessage { ... }
```

**Updated `core/erase.go`:**

```go
func Erase[In, Out any](t Tool[In, Out]) AnyTool {
    meta := t.Definition()
    schema := deriveSchema[In]()  // panics on unsupported types
    return &erasedTool[In, Out]{
        tool: t,
        def: ToolDefinition{
            Name:        meta.Name,
            Description: meta.Description,
            Parameters:  schema,
        },
    }
}
```

The existing `erasedTool[In, Out].ExecuteRaw` and `Name()`/`Definition()` accessors stay (`AnyTool.Definition()` returns the captured `def` instead of calling through to `t.Definition()`).

**Updated `core/tool.go`:**

```go
type ToolMeta struct {
    Name        string
    Description string
}

type Tool[In, Out any] interface {
    Definition() ToolMeta
    Execute(ctx context.Context, in In) (Out, error)
}

type StreamingTool[In, Out any] interface {
    Tool[In, Out]
    ExecuteStream(ctx context.Context, in In, ch chan<- StreamEvent) (Out, error)
}
```

Reflection internals (private helpers in `core/schema.go`): walk `reflect.Type` of `In`, emit `map[string]any` then `json.Marshal` it. Field iteration honors `json` tags for naming and `omitempty`, pointer-typed fields are optional, `describe`/`enum` tags are read for properties. The type-assertion check for `SchemaProvider` happens first; if the type implements it, return that raw bytes unchanged.

**Testing approach:**
- Table-driven tests in `core/schema_test.go` covering every supported type, every tag, every rejection case.
- One end-to-end test per migrated tool comparing emitted schema against the previously-hand-written JSON Schema (modulo formatting). These double as living examples for the migration guide.
- Panic-message tests for each rejected case verifying the message names the field, type, and supported alternatives.

## Provider-compatibility subset

The reflector emits the JSON Schema features accepted by **all three current providers** (Gemini, OpenAI, Anthropic):
- `type`
- `properties` / `required`
- `items`
- `additionalProperties`
- `enum`
- `description`
- `format` â€” only for `time.Time` â†’ `"date-time"`

**Deliberately not emitted by reflection:** `oneOf`, `anyOf`, `allOf`, `$ref`, `not`, `if`/`then`/`else`. Any tool needing these uses the `JSONSchema()` escape hatch and accepts that its schema may not work with every provider.

This subset is documented in code (top-of-file comment in `core/schema.go`) and in the user-facing docs.

## Testing approach

- **Reflector unit tests** are the primary safety net. Every supported type, every tag, every rejection.
- **Equivalence tests** for each migrated tool â€” emit the new schema, compare to the pre-migration hand-written one. Any non-trivial difference is a bug in the migration or the reflector.
- **End-to-end tests** in `agent/` for at least one migrated tool exercised through a real loop run with a fake provider.
- **CI gate**: `go test ./...` from root and from every satellite must pass before merge. `golangci-lint run ./...` must pass.

## Open questions

None blocking implementation. Two notes for the implementation plan:

1. **Built-in tool defs in `agent/`.** `askUserToolDef`, `executePlanToolDef`, `spawnAgentToolDef` build `ToolDefinition` values directly. The implementation plan should decide whether they migrate to typed input structs (preferred â€” consistency) or call `deriveSchema` directly via a small helper. Lean: typed structs even for built-ins, with their own `Definition() ToolMeta`-style helpers.

2. **Two-pass struct walking for recursion detection.** Naive reflection on a recursive type infinite-loops. The reflector needs a visited-types set per `deriveSchema` call. The plan should specify the recursion-detection algorithm explicitly.
